```

```


#### B. `output/en/` - 정리된 영어 SRT
**목적**: 후처리를 거친 깨끗한 영어 자막

**특징**:
- 코드 수정 시 **매번 새로 생성**됨
- raw 파일을 기반으로 후처리 적용
- 번역의 입력 소스로 사용

**후처리 과정**:
1. **긴 블록 분할**: 8초 초과 블록을 문장 단위로 분할
2. **필러 제거**: "Uh.", "Um.", "Hmm." 등 의미 없는 블록 삭제
3. **불완전 문장 병합**: 전치사/접속사로 끝나는 블록을 다음 블록에 합침
4. **화자 전환 분리**: 질문→답변 패턴, 시간 갭, 응답 시작 패턴 감지
5. **문장 병합**: 한 블록 = 한 문장으로 정리

**결과**:
- 완벽한 문장 구조
- 화자 전환이 명확하게 분리
- 번역하기 좋은 형태

**예시**:
```
1
00:00:00,000 --> 00:00:15,720
We did recently switch our version of Teams, so I just need to find how it works now. This is on my end, not yours, so you don't have to worry. But for now, let me share my screen.
```

## 3. 원인 분석

### Whisper API의 전사 특성

#### A. 음성 휴지(pause) 감지
```
화자: "because I'm querying for title and name..." [0.5초 쉼]
Whisper: "because I'm querying for title and name." ← 온점 추가
```
- 화자가 말하다가 잠깐 멈추면 Whisper는 "문장이 끝났다"고 판단
- 실제로는 생각하면서 말하는 중이지만, AI는 휴지를 문장 종결로 해석

#### B. 억양 변화 (Intonation)
- 화자가 말 끝을 내리면 (falling intonation) → 온점(.)
- 화자가 말 끝을 올리면 (rising intonation) → 물음표(?)
- "name." 부분에서 화자가 목소리를 내렸을 가능성

#### C. 구어체 처리 한계
- Whisper는 "완결된 문장"으로 학습됨
- 구어체의 불완전한 문장, 말 더듬기, 중간에 끊기는 패턴을 잘 처리하지 못함
- 특히 회의/강의 같은 즉흥 발화는 문장 구조가 불완전한 경우가 많음

#### D. 실제 발화 패턴
```
화자: "No, so in this query, because I'm querying for title and name..."
      [0.5초 생각]
      "Uh, this line 11 essentially..."
      [0.3초 생각]
      "Isn't doing anything aside from one important fact, which is..."
      [0.4초 생각]
      "Suppose there's an option..."
```

화자는 **하나의 긴 설명**을 하고 있지만, 중간중간 생각하면서 말하기 때문에 Whisper는 각 휴지 지점을 문장 끝으로 판단합니다.

### 기존 코드의 한계

**기존 `_clean_whisper_artifacts` 함수**:
- 전치사/접속사로 끝나는 경우만 병합
- "because I'm querying for title and name." 같은 경우는 "name"이 명사로 끝나서 패턴에 걸리지 않음
- 결과: 병합되지 않고 독립된 문장으로 남음

#### 휴리스틱 1: 종속절 감지

**코드**:
```python
_SUBORDINATE_START = re.compile(
    r'^\s*(because|since|although|though|while|if|when|where|unless|until|as|after|before)\b',
    re.I,
)

starts_with_subordinate = bool(_SUBORDINATE_START.match(text))
```

**동작**:
- 블록이 `because`, `since`, `although` 등으로 시작하면 불완전한 문장으로 판단
- 온점으로 끝나도 다음 블록과 병합

**적용 예시**:
```
Before: "because I'm querying for title and name." (독립 블록)
After:  "because I'm querying for title and name, uh, this line..." (병합)
```

#### 휴리스틱 2: 관계절 감지

**코드**:
```python
_INCOMPLETE_CLAUSE = re.compile(
    r'\b(which is|that is|who is|what is|where is|how is|why is)\s*[.!?,]?\s*$',
    re.I,
)

ends_with_incomplete_clause = bool(_INCOMPLETE_CLAUSE.search(text))
```

**동작**:
- 블록이 `which is`, `that is` 등으로 끝나면 불완전한 문장으로 판단
- 명백히 뒤에 내용이 이어져야 함

**적용 예시**:
```
Before: "aside from one important fact, which is." (독립 블록)
After:  "aside from one important fact, which is, suppose there's..." (병합)
```

#### 휴리스틱 3: 시간 갭 감지

**코드**:
```python
SHORT_GAP_MS = 500  # 0.5초

time_gap_ms = next_block.start.ordinal - current_block.end.ordinal

```

**동작**:
- 블록 간 시간 간격이 0.5초 미만이고
- 다음 블록이 소문자로 시작하면
- 같은 문장으로 간주하여 병합

**적용 예시**:
```

블록 B: "uh, this line..." ← 소문자 시작
→ 병합
```

**코드**:
```python
_LOWERCASE_OR_FILLER_START = re.compile(
    r'^[a-z]|^(uh|um|ah|oh|er)',

```

**동작**:
- 다음 블록이 소문자나 필러(`uh`, `um`)로 시작하면
- 이어지는 문장으로 판단하여 병합

**적용 예시**:
```

블록 B: "uh, this line..." ← 필러로 시작

```

#### 휴리스틱 5: 화자 전환 감지 (분할)

**코드** (5/8 개선 후):
```python
_SPEAKER_SPLIT = re.compile(
    r'(?<=[.!?,])\s+(?=(?:Yeah|Yes|No|Okay so|Thanks|Thank you|'
    r'That\'s great|Absolutely|Exactly|Right so|All right))',
    re.I,
)
```

**동작**:
- **쉼표/마침표 뒤에서만** 발동 (공백만으로는 불가)
- "Yeah", "Yes", "No" 등 **명확한 응답어**에서만 분할
- `but`, `I think`, `sure` 등 일반 접속사에서는 **분할하지 않음**

**적용 예시**:
```
Before: "...that concern heard. Yeah so the next thing..."
After:  
  블록 1: "...that concern heard."
  블록 2: "Yeah so the next thing..."
```

**⚠️ 5/8 개선 포인트**: 기존 정규식은 `but`, `I think`, `sure`, `that` 등 14개 일반 단어에서도 분할하여 한 문장이 3~6조각으로 분해되는 버그가 있었음. 명확한 응답어 10개만 남기고 대폭 축소함.

## 5. 프롬프팅 전략 (함수별 상세 설명)

### 5.1 시스템 프롬프트 생성

**파일**: `C:\Users\gksmf\SRTLinker\prompts.py`  
**함수**: `build_system_prompt(target_lang, glossary_path)` (Line 6~30)

**목적**: GPT에게 "전문 영상 자막 번역가" 역할 부여 및 12가지 규칙 전달


**코드 구조**:
```python
def build_system_prompt(target_lang: str, glossary_path: Path | None = None) -> str:
    # 1. 용어집 로드
    glossary_block = ""
    if glossary_path and glossary_path.exists():
        data = json.loads(glossary_path.read_text(encoding="utf-8"))
        keep = data.get("keep_as_is", [])  # 영문 유지 단어
        fixed = data.get("fixed_translations", {})  # 고정 번역
    
    # 2. 시스템 프롬프트 생성
    return f"""
    너는 전문 영상 자막 번역가이다. 각 블록은 한 개의 완전한 문장이고, 이를 자연스러운 {target_lang}로 번역한다.
    
    [반드시 지킬 규칙]
    1) translate 배열의 각 블록만 번역하고 context_before / context_after는 문맥 참고용이다
    2) 출력의 id 집합과 순서는 입력과 정확히 일치해야 한다
    3) 한 블록 = 하나의 번역
    4) 의역 허용, 직역보다 의미 기반 자연스러운 표현 우선
    5) 회의/교육 톤에 맞는 경어 반말(~합니다 / ~입니다)을 사용한다
    6) 기술 용어(Anzo, Ontology, Dataset, Knowledge Graph, RDF, SPARQL, API, SDK, JSON 등)는 영문 그대로 유지한다
    7) 고유명사/제품명/버전명은 원문 표기 유지
    8) 이모티콘, 따옴표, 마크다운 서식 등 원본 서식은 의미 있을 때만 유지한다
    9) 구어체(uh, um, you know, like 등 필러) 는 자연스러운 범위 내에서 제거 가능
    10) 줄바꿈(\\n)은 원문 구조가 자연스러운 경우에만 사용
    11) 원문이 쉼표(,)로 이어지는 문장이면 번역문도 쉼표로 이어야 한다
    12) 출력은 반드시 지정된 JSON 스키마 {{items:[{{id, text}}]}} 를 따른다
    
    [용어집]
    {glossary_block}
    """
```


**12가지 핵심 규칙 상세 설명**:

| 번호 | 규칙 | 목적 | 예시 |
|------|------|------|------|
| 1 | translate만 번역 | context는 참고용 | context_before는 번역 안 함 |
| 2 | id 정확히 일치 | 블록 매핑 보장 | 입력 id [1,2,3] → 출력 id [1,2,3] |
| 3 | 한 블록 = 하나의 번역 | 블록 경계 유지 | 블록 합치거나 나누지 않음 |
| 4 | 의역 허용 | 자연스러운 표현 | "I think" → "생각합니다" (직역 X) |

**규칙 11번이 중요한 이유**:
- Whisper가 만든 SRT에서 한 문장이 여러 블록으로 쪼개져 있을 때
- 각 블록을 독립적으로 번역하면 GPT가 쉼표로 이어지는 문장을 마침표로 끊어버리는 문제 발생
- 이 규칙을 추가해서 원문의 부호 구조를 유지하도록 강제

### 5.2 번역 요청 페이로드 생성

**파일**: `C:\Users\gksmf\SRTLinker\prompts.py`  
**함수**: `build_user_payload(translate_blocks, context_before, context_after, target_lang)` (Line 33~45)

**목적**: GPT에게 전달할 번역 요청 데이터 구조화


**코드 구조**:
```python
def build_user_payload(translate_blocks, context_before, context_after, target_lang):
    payload = {
        "target_language": target_lang,  # "Korean"
        "context_before": context_before,  # 앞 5개 블록 (참고용)
        "translate": translate_blocks,     # 번역할 30개 블록 (실제 대상)
        "context_after": context_after,    # 뒤 5개 블록 (참고용)
        "instruction": "translate 배열의 각 블록을 target_language로 번역하고, {items:[{id,text}]} 형태의 JSON으로만 응답하라. id는 입력과 정확히 동일해야 한다."
    }
    return json.dumps(payload, ensure_ascii=False)
```

**페이로드 구조 상세**:

| 필드 | 타입 | 설명 | 예시 |
|------|------|------|------|
| `target_language` | string | 번역 대상 언어 | "Korean" |
| `context_before` | array | 앞 5개 블록 (참고용, 번역 안 함) | [{id:5, text:"..."}, ...] |
| `translate` | array | 번역할 30개 블록 (실제 번역 대상) | [{id:6, text:"..."}, ...] |
| `context_after` | array | 뒤 5개 블록 (참고용, 번역 안 함) | [{id:36, text:"..."}, ...] |
| `instruction` | string | 번역 지시사항 | "translate 배열의..." |

**문맥 제공 메커니즘**:
```
블록 1~5   ← context_before (참고용)
블록 6~35  ← translate (번역 대상)
블록 36~40 ← context_after (참고용)
```

**왜 30개씩 청크로 나누나?**
1. **API 토큰 제한**: GPT-4o는 한 번에 처리할 수 있는 토큰 수 제한
2. **효율성**: 30개씩 나누면 병렬 처리 가능
3. **안정성**: 일부 청크 실패 시 해당 청크만 재시도

**왜 앞뒤 5개 블록을 문맥으로 제공하나?**
1. **자연스러운 번역**: 앞뒤 문맥을 알아야 자연스러운 번역 가능
2. **대명사 해석**: "it", "this", "that" 등의 지시 대상 파악
3. **화자 전환 인식**: 누가 말하는지 파악하여 적절한 어투 선택

### 5.3 번역 안정성 보장

**파일**: `C:\Users\gksmf\SRTLinker\translator.py`  
**함수**: `translate_chunk(chunk, max_retry_missing=1)` (Line 82~120)

**목적**: 청크 단위 번역 + 자동 재시도 + 원문 보존


**코드 구조**:
```python
def translate_chunk(self, chunk, max_retry_missing=1):
    expected_ids = [b.id for b in chunk.translate]  # [1, 2, 3, ..., 30]
    id_to_block = {b.id: b for b in chunk.translate}
    
    # 1단계: 전체 청크 번역 시도
    try:
        got = self._call_api(chunk.translate, chunk.context_before, chunk.context_after)
    except Exception as e:
        log.warning("API call failed for chunk: %s", e)
        got = {}
    
    # 2단계: 누락된 id만 다시 요청
    missing = [i for i in expected_ids if i not in got]
    if missing:
        log.info("Retrying missing ids %s", missing)
        sub_blocks = [id_to_block[i] for i in missing]
        try:
            more = self._call_api(sub_blocks, chunk.context_before, chunk.context_after)
        except Exception as e:
            log.warning("Retry call failed: %s", e)
            more = {}
        got.update({i: t for i, t in more.items() if i in id_to_block})
    
    # 3단계: 최종 결과 (누락 시 원문 유지)
    result = {}
    fallback_ids = []
    for i in expected_ids:
        if i in got:
            result[i] = got[i]
        else:
            result[i] = id_to_block[i].text  # 원문 유지
            fallback_ids.append(i)
    
    if fallback_ids:
        log.warning("Fell back to original text for ids: %s", fallback_ids)
    
    return result
```

**3단계 안정성 메커니즘**:

| 단계 | 동작 | 실패 시 처리 | 목적 |
|------|------|--------------|------|
| 1 | 전체 청크 번역 | 빈 결과 반환 | 정상 케이스 처리 |
| 2 | 누락 id만 재요청 | 빈 결과 반환 | 부분 실패 복구 |
| 3 | 원문 유지 | 원문 그대로 저장 | 완전 실패 대응 |

**추가 안정성 장치**:

1. **JSON Schema Strict 모드**:
```python
JSON_SCHEMA = {
    "name": "srt_translation",
    "schema": {

        "properties": {
            "items": {
                "type": "array",
                "items": {
                    "type": "object",
                    "properties": {
                        "id": {"type": "integer"},
                        "text": {"type": "string"}
                    },
                    "required": ["id", "text"]

            }
        },

}
```
- GPT 응답을 `{items:[{id, text}]}` 형식으로 강제
- id 매핑 오류 방지

2. **Exponential Backoff 재시도**:

    wait=wait_exponential(multiplier=1, min=1, max=5),
    retry=retry_if_exception_type(Exception)

## 9. 금일(5/8) 추가 개선사항

### 9.1 화자 전환 패턴 대폭 축소
- 기존: `but`, `I think`, `sure`, `that` 등 14개 일반 단어에서 분할 → **오분할 빈발**
- 개선: `Yeah`, `Yes`, `No`, `Thanks` 등 명확한 응답어 10개만 유지
- 발동 조건 강화: 공백(`\s`) → **문장부호(`.!?,`) 뒤에서만**

### 9.2 문장 분할 정밀화
- 기존: `(?<=[.!?])\s+` → "2.5 million" 등 숫자 뒤에서도 분할
- 개선: `(?<=[.!?])\s+(?=[A-Z])` → **대문자 시작**일 때만 분할

### 9.3 불완전 문장 병합 제한 완화
- 기존: 모든 블록 병합 최대 15초
- 개선: 마침표 없는 **불완전 문장**은 **30초**까지 병합 허용
- 사례: "...feel" + "free to use..." (15.2초) → 이제 정상 병합됨

### 9.4 파이프라인 안정성
- **en.srt 재사용**: 기존 영어 SRT가 있으면 전사 건너뜀 (수동 편집본 보존)
- **부분 저장**: 전사 중간 실패 시 완료된 부분까지 저장 (전체 유실 방지)
- **에러 가시화**: 번역 실패 시 콘솔에 즉시 표시 (기존: 조용히 영어로 대체)
- **이중 처리 제거**: raw → 후처리가 2번 실행되던 버그 수정
